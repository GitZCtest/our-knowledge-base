
> 适用于各种任务的标准化执行流程

## 核心理念

**任务完成 = 清晰定义 × 合理拆解 × 持续推进 × 及时调整**

这个方法论适用于：学习任务、项目开发、副业探索、日常事务等各种场景。

---

## 第一步：任务定义与澄清（10-15分钟）

### 1.1 明确"什么是完成"

用**验收标准**替代模糊目标：

| ❌ 模糊目标 | ✅ 清晰验收标准 |
|----------|-------------|
| "学会C++文件操作" | "能不查文档写出读取文件、遍历目录的代码" |
| "做个小红书账号" | "发布15条笔记，获得100+粉丝" |
| "提升算法能力" | "独立AC 50道中等题，能向别人讲清楚思路" |

**关键问题**：
- 做到什么程度算完成？
- 有没有可量化的指标？
- 如果让别人检查，他们怎么判断你做完了？

### 1.2 识别任务类型

不同类型任务，策略不同：

| 任务类型    | 特征      | 执行要点           |
| ------- | ------- | -------------- |
| **学习型** | 掌握知识/技能 | 重视理解+练习，需要反馈循环 |
| **创造型** | 产出作品/内容 | 允许迭代，先完成再完美    |
| **执行型** | 完成固定流程  | 列清单，逐项划勾       |
| **探索型** | 验证假设/试错 | 设定小实验，快速反馈     |

### 1.3 评估约束条件

**时间约束**：
- 总共有多少时间？
- 每天能投入多少？
- 有没有deadline？

**资源约束**：
- 需要哪些工具/材料？
- 需要别人协作吗？
- 有没有现成参考？

**能力约束**：
- 当前能力够不够？
- 需要先补什么前置知识？
- 哪些部分可以求助AI/他人？

## 第二步：任务拆解与规划（20-30分钟）

### 2.1 逆向拆解法

从验收标准倒推：

```
最终交付物 → 需要哪些部分 → 每部分需要哪些步骤 → 每步需要多久
```

**示例**（以"完成代码统计工具"为例）：

```
层级1：最终交付物
└─ 可运行的代码统计工具

层级2：需要哪些部分
├─ 参数解析模块
├─ 文件遍历模块
├─ 代码分析模块
└─ 报告生成模块

层级3：每部分需要哪些步骤（以参数解析为例）
├─ 定义命令行接口
├─ 实现参数解析逻辑
├─ 添加错误处理
└─ 写测试用例

层级4：每步需要多久
├─ 定义接口（30分钟）
├─ 实现逻辑（2小时）
├─ 错误处理（1小时）
└─ 测试（1小时）
```

### 2.2 识别关键路径

**找出必须按顺序做的部分**：
- A必须在B之前完成吗？
- 哪些可以并行？
- 哪个是瓶颈？

**优先级排序**：
1. **必须做且有依赖关系的**（关键路径）
2. **必须做但可以灵活安排的**
3. **锦上添花的**（可选）

### 2.3 设定检查点

**大任务：每3-5天一个检查点**
**小任务：每天一个检查点**

检查点的作用：
- 确认进度是否符合预期
- 及时发现问题并调整
- 提供成就感（完成阶段性目标）

## 第三步：执行与推进

### 3.1 每日执行流程

#### 开始前（5分钟）
- [ ] 今天要完成的具体任务是什么？
- [ ] 预计需要多长时间？
- [ ] 需要哪些工具/资料？

#### 执行中
**使用"番茄工作法"变体**：
- **前15分钟：独立思考/动手**
  - 不开AI，先自己想
  - 写伪代码、画草图、列思路
  - 目的：避免直接依赖AI
  
- **之后：高效推进**
  - 卡住时分级求助（文档 → AI思路 → AI具体帮助）
  - 保持专注，避免分心
  
- **每小时休息5-10分钟**

#### 结束后（10分钟）
- [ ] 今天完成了什么？
- [ ] 遇到什么问题？怎么解决的？
- [ ] 明天要做什么？

### 3.2 处理"卡住"的情况

**分级求助策略**：

```
Level 1: 自己想5分钟
  ↓ 还是卡住
Level 2: 查文档/搜索（10分钟）
  ↓ 还是卡住
Level 3: 问AI"这个问题通常怎么解决"（要思路）
  ↓ 还是卡住
Level 4: 问AI"帮我实现"（要代码）
  ↓ 还是卡住
Level 5: 跳过，做其他部分，回头再攻克
```

**重要**：Level 4（让AI直接写代码）之后，必须：
- 逐行读懂代码
- 尝试不看代码自己重写一遍核心逻辑
- 记录"为什么之前卡住了"

### 3.3 记录与沉淀

**每日记录**（在memos）：

```markdown
## Day X - [日期] - [任务名称]

### 今日目标
- 

### 实际完成
- 

### 卡住的地方 & 解决方法
- 

### 学到的新东西
- 

### 明天计划
- 
```

**每周整理**（到obsidian）：
- 提炼可复用的方法/模式
- 更新到"学习方法实验室"

## 第四步：检查点复盘与调整

### 4.1 检查点复盘（每3-5天）

**三个核心问题**：
1. **进度符合预期吗？**
   - 提前 → 可以加点可选内容
   - 正常 → 继续保持
   - 落后 → 分析原因，调整计划

2. **遇到什么意外？**
   - 技术难点超出预期
   - 时间估算不准
   - 外部干扰

3. **需要调整什么？**
   - 优先级重排
   - 砍掉不重要的部分
   - 寻求额外帮助

### 4.2 动态调整原则

**允许调整的**：
- 具体实现方式
- 时间分配
- 子任务优先级
- 可选功能的取舍

**不轻易调整的**：
- 核心验收标准
- 最终deadline
- 任务的根本目标

---

## 第五步：完成与沉淀

### 5.1 验收确认

**对照最初的验收标准**：
- [ ] 每一条都达成了吗？
- [ ] 有没有遗漏的地方？
- [ ] 如果有人用/看，能正常工作吗？

### 5.2 复盘总结（30-60分钟）

**四个维度**：

1. **产出质量**
   - 完成度如何？
   - 有什么亮点？
   - 有什么不足？

2. **过程效率**
   - 时间估算准吗？
   - 哪个环节最耗时？
   - 有没有走弯路？

3. **方法有效性**
   - 哪些方法好用？
   - 哪些方法不行？
   - 下次可以怎么改进？

4. **能力提升**
   - 学到了什么新东西？
   - 对什么有了新认识？
   - 下次遇到类似任务会怎么做？

### 5.3 沉淀可复用资产

**代码/模板**：
- 好用的代码片段
- 项目结构模板
- 配置文件

**方法/流程**：
- 更新到"学习方法实验室"
- 形成"XXX任务执行模板"

**经验/教训**：
- 记录到对应主题笔记
- 标记"下次注意"

---

## 关键原则总结

1. **先定义清楚"完成"，再开始做**
2. **大任务必须拆解，小到能在1-2小时完成**
3. **留出独立思考时间，不立刻依赖AI**
4. **设定检查点，及时调整**
5. **每次完成后都要复盘和沉淀**
